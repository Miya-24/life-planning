# Issue 作成 Strategy

このドキュメントは、GitHub Issue を  
「行動ログ」として正しく機能させるための方針を定義する。

---

## 1. Issue の本質

> **Issue = 実施した作業の記録（ログ）**

Issue は「理解できたか」「うまくできたか」を測るものではなく、  
**実際にその作業をやったかどうか**を記録するための単位とする。

---

## 2. Close / Done のルール

- **理解できたかどうかに関わらず**
- **決めた作業を実施したら**

→ Project 上で `Done` に移動し、Issue を `Close` する。

理解度・出来・感想は Issue のコメントや週次レビューで扱う。

---

## 3. Issue の粒度

- 1 Issue = 1回の作業で完了できる単位（目安：30〜90分）
- 平日用：30〜60分
- 土曜用（long）：90分

---

## 4. Title のルール

- 何をどれくらいやるかが一目で分かること
- 作業量が分かる情報を含める

例：
- 英語シャドーイング（5分 × 3日）
- AP 午前 ネットワーク（問1–20）
- AZ900 Learn: Core Azure services

---

## 5. Description の形式

すべての Issue は次の形式で記載する。

**目的（Why）**  
- なぜこの Issue を行うのか

**やること（What）**  
- 実際に行う作業を箇条書きで記載

**完了条件（Evidence）**  
- **実施したことが確認できるチェック項目**

---

## 6. 完了条件のルール

完了条件は必ず  
作成したテンプレート（heavy.md / light.md）をそのまま使い、余計なことは書かなくて良い。

理解度や感覚は完了条件に含めない。

---

## 7. Milestone と Project との関係

すべての Issue は、次の2つに必ず紐づける。

### ① Milestone（長期目標）
Issue は必ず、以下のいずれかの Milestone に属する。

- 応用情報（AP）
- Azure Fundamentals
- 英語
- ジム
- テニス
- ベース

これにより、**日々の作業が長期目標と直結**する。

---

### ② Project（週の計画）
Issue は必ず、その週の GitHub Project に割り振る。

- Project は 1週間につき1つ作成
- カラム：Backlog / In Progress / Done

これにより、**週単位での進捗管理と振り返り**が可能になる。

---

## 8. 進捗管理の考え方

- Close された Issue = 実際に行った作業量
- 週の進捗 = その週の Project で Done に入った Issue
- 長期の進捗 = Milestone ごとの Issue 完了数

理解・出来・反省は  
- Issue コメント  
- 週間振り返り  
で扱う。

Issue はあくまで **実行ログ** として使う。
