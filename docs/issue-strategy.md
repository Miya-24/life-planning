# Issue 作成方針

このドキュメントは、GitHub Issue を作る際の方針をまとめたものです。  
主に **目的・やること・完了条件** の形式で Issue を作ることで、学習や習慣を効率的に管理します。

---

## 1. Issue の粒度

- **1 Issue = 1回の作業で完了できる粒度**（目安：30〜90分）  
- 大きすぎるタスクは小分けにする  
- 小さすぎるタスクはまとめて管理する  

---

## 2. Title の作り方

- 何をいつまでにどのくらいやるかが一目で分かる  
- 日数や回数を入れると習慣化がわかりやすい  

### 例
- 英語シャドーイング(5分×3日)  
- AZ900 Learn: Describe core Azure services  
- English Grammar in Use: Units 1 - 3  

> **ポイント**：Title を見ただけで「何をこなすべきか」が理解できることが重要

---

## 3. Description の形式

すべての Issue は次の形式で記載する。

### 形式

**目的：**  
- なぜこの Issue を行うのか、学習の狙いや達成したいことを簡潔に書く

**やること：**  
- 実際に行う作業内容を箇条書きで書く  
- 具体的に書くほど進捗管理がしやすい

**完了条件：**  
- 何をもって「完了」と判断するか明確に書く  
- できれば **定量化できる成果物や理解レベル** を設定する

---

### 例1
**Title:** 
英語シャドーイング(5分×3日)  

**Description:**  
**目的：** 
- 英語の発音とリスニング力を向上させる
  
**やること：**  
- 1日5分、短い英文をシャドーイング  
- 発音・リズムを意識する  
- 3日間連続で行う
  
**完了条件：**  
- 3日間すべてシャドーイングを完了  
- 簡単なフレーズを聞いて即口に出せるようになる  

### 例2
**Title:** 
AZ900 Learn: Describe core Azure services  

**Description:**  
**目的：**
- Azure の主要サービスの概要を理解する  

**やること：**  
- Microsoft Learn モジュール「Describe core Azure services」を学習  
- 主要サービスの特徴・用途を整理する  

**完了条件：**  
- モジュールを完了し、サービスの概要を説明できる  

---

## 4. 管理上のポイント

- Issue は **作業の実行可能性を最優先**  
- 日々の進捗は Issue の完了数で管理  
- 長期目標と紐づける（Milestone）  
- 週単位の Project に割り振る  

---

## 5. 注意点

- **曖昧なタイトルや目的は避ける**  
- 完了条件を明確にすることで、達成感と振り返りがしやすくなる  
- 過去問や参考書、オンライン学習は URL を Description に添付すると便利
