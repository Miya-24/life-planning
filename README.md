## 全体運用フロー（このリポジトリの使い方）

このリポジトリは  
**「戦略 → 週次計画 → 日々の実行 → 振り返り」**  
を GitHub 上で回すための個人用運用システムです。

---

## 1. 戦略（Strategy）

### 役割
- 何を優先するか
- どういう判断基準で進めるか
- 迷ったときの拠り所

### 場所
- `docs/` 配下の各 `*_strategy.md`

### ポイント
- 戦略は頻繁に変えない
- 実行がうまくいかない場合は「自分」ではなく「戦略や運用」を見直す

---

## 2. スケジュール（Schedule）

### 役割
- 平日・週末の時間枠を固定する
- heavy / light の配置を決める

### 場所
- `schedules/` 配下の md

### ポイント
- 曜日ごとの heavy / light 枠は固定
- Issue の内容だけを入れ替える

---

## 3. 週次計画（Project）

### 役割
- その週にやることを可視化する

### 運用
- 1週間につき 1つ GitHub Project を作成
- カラムは以下の3つのみ
  - Backlog
  - In Progress
  - Done

### ポイント
- その週にやる Issue はすべて Project に入れる
- 週が終わったら Project は閉じる

---

## 4. 実行（Issue）

### 役割
- 実際にやる作業の単位（実行ログ）

### ルール
- 1 Issue = 30〜90分で完了できる作業
- 完了条件は「理解」ではなく **実施した証拠**
- 実施したら Done / Close（出来・理解度は問わない）

### 種類
- `heavy`：新規理解・思考が必要な作業
- `light`：復習・習慣・負荷の軽い作業

---

## 5. 週末レビュー（Weekly Review）

### 役割
- 1週間を閉じ、次週につなげる

### 書く場所
- GitHub Issue（週間レビュー用 Issue）の **Add a comment**

### テンプレ
- `docs/review_template.md`

### ポイント
- Done の Issue を事実として見る
- 反省よりも「次にどうするか」を重視
- 次週の Issue 案を必ず1つ以上出す

---

## 6. この運用で大事にしていること

- 気合ではなく仕組みで回す
- できなかったら自分を責めない
- Close = 成功 ではない  
  Close = **実行した記録**
- 続いている状態そのものが成果

---

## まとめ

- 戦略は `docs/`
- スケジュールは `schedules/`
- 実行は Issue
- 週は Project
- 振り返りは Issue コメント

この流れを淡々と回す。
